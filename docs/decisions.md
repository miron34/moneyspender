## 2026-04-28 — История: иконки сортировки/поиска в правом углу + фильтр диапазона

**Контекст:** В MVP-версии Истории были два pill'а — «Категория» (фильтр) и «Дата» (на самом деле — сортировка `по дате/по сумме`). Имя «Дата» сбивало с толку. Также не было возможности отфильтровать траты по диапазону дат.

**Выбор:**
- В правом верхнем углу — две иконки: 🔍 поиск + ↕ сортировка. Иконка сортировки меняет вид (clock / bar-chart-2) в зависимости от выбранного режима. Активный режим подсвечен accent-цветом.
- Pill «Дата» убран. На его месте — pill «Период» (фильтр по диапазону дат), открывает новый `DateRangeSheet`.
- Поиск разворачивается inline-строкой под фильтрами (TextInput с `autoFocus`). Live-фильтр по `expense.name` (case-insensitive). При закрытии query очищается.
- Порядок применения фильтров: категория → диапазон дат → поисковый запрос → сортировка.

**Почему поиск inline, а не в шите:** Поиск — частая операция, не должен прятаться за двумя тапами. Inline-строка с автофокусом даёт мгновенный ввод.

**Почему диапазон дат — отдельный шит, а не два инпута:** Drum-пикер (как у `DatePickerSheet`) — наш единый паттерн ввода даты. Два drum'а друг под другом дают сразу обе границы. Кнопка «Сбросить» очищает оба значения.

**Альтернатива (отклонили):** иконка фильтра-всё-в-одном с шитом из 4 секций (категория/период/сортировка/поиск). Слишком много действий за одним жестом, теряется UX-разделение «вижу что фильтрую» от «как сортирую».

## 2026-04-28 — Drill-down с чартов: tap по сегменту/бару → История

**Контекст:** Чарты `BarChart` и `DonutChart` были чисто декоративные. Аналитика и Главная не давали способа быстро провалиться в траты конкретной категории.

**Выбор:** У обоих чартов добавлен опциональный проп `onSelect(catId)`. Если передан — каждый бар/сегмент становится кликабельным:
- **Хаптика**: `lightTap()` на press-in.
- **Pulse-анимация**: бар — scale 1 → 1.06 → 1; сегмент — strokeWidth 22 → 28 → 22. Easing — тот же `cubic-bezier(0.16, 1, 0.3, 1)`, что и у анимации появления.
- **Навигация**: `setTimeout(140 ms)` — даёт пользователю увидеть pulse, после чего `router.push('/history', { cat })`.

В Истории: `useLocalSearchParams<{ cat?: string }>()` + `useEffect` ставит `filterCat` и сразу вызывает `router.setParams({ cat: undefined })`, чтобы при ручном переключении табов фильтр не возвращался.

**DonutChart — два способа тапа по сегменту:**
1. **Hit-test по SVG**: на `Pressable` поверх `<Svg>` ловим `locationX/Y`, считаем угол `atan2` от центра, ищем slice по диапазону `[startAngle, endAngle]`. Корректно учитывается `rotate(-90)` (SVG starts at 12 o'clock).
2. **Тап по легенде**: каждый ряд легенды обёрнут в `Pressable`. Запускает ту же pulse-анимацию сегмента и навигацию.

**Почему оба способа:** Hit-test даёт «настоящий» тап по сегменту, как пользователь ожидает. Легенда — резервный способ для маленьких сегментов, куда трудно попасть пальцем (< 5%). Реализация дешёвая — оба используют одну и ту же функцию `triggerSlice(id)`.

**Альтернатива (отклонили):** только тап по легенде (D2). Проще, но формально не решает задачу «тап по сегменту» из требования.

## 2026-04-27 — Web-deploy: Vercel + cleanUrls

**Контекст:** MVP готов, нужна публичная ссылка для тестирования и шаринга. Стек предполагает Vercel для веб-версии.

**Выбор:** Vercel Hobby (бесплатный) + auto-deploy с GitHub `main`. Конфиг `vercel.json` фиксирует build/output команды.

**Почему Vercel:**
- Лучшая интеграция с Expo Web (auto-detect не работает, но конфиг минимальный)
- Auto-deploy с push в main за ~2 мин
- Глобальный CDN, HTTPS из коробки
- Бесплатный тариф достаточен для одного пользователя (100 GB трафика / мес)

**Почему `cleanUrls: true`:** Expo экспортирует плоские `history.html` / `analytics.html` / `profile.html` в `dist/`. Vercel по умолчанию не подставляет `.html` при URL без суффикса → запрос `/history` возвращал 404. С `cleanUrls=true` Vercel автоматически делает маппинг.

**Альтернативы (отклонили):**
- *Netlify* — тот же класс инструмента, но Vercel ближе к React-экосистеме
- *Cloudflare Pages* — есть, но менее удобный UI для preview-deploys
- *GitHub Pages* — бесплатно, но требует ручной настройки CI и не даёт preview

## 2026-04-27 — Auth и RLS отложены до post-MVP

**Контекст:** В текущей версии БД доступна по anon-ключу любому, кто знает URL Supabase-проекта. Все пишут в одну БД. Это намеренно.

**Выбор:** Не вводить auth и RLS до тех пор, пока приложением реально не пользуется кто-то кроме автора.

**Почему:**
- MVP — личный проект одного пользователя ("Мирон"). Auth добавляет 5+ экранов и логики ради защиты от риска, которого по факту нет.
- Anon-ключ публичный по дизайну Supabase, его не надо прятать.
- При публикации на других пользователей шаг E из roadmap покрывает это: добавить `user_id` колонку, включить RLS, политики `auth.uid() = user_id`.

**Риск:** если кто-то узнает URL+anon-ключ из исходников Vercel-сборки, может писать/удалять данные в БД. Принимаем — это не production с сотнями пользователей.

## 2026-04-25 — Стек MVP: Expo + Expo Router + Zustand + Reanimated v4

**Контекст:** Целевые платформы — iOS, Android и web. Web-режим нужен для отладки в браузере (iPhone-фрейм 390×844). Один пользователь, без auth в MVP.

**Выбор:**
- React Native + Expo SDK 54
- Expo Router v6 (file-based)
- Zustand для стейта
- Reanimated v4 для анимаций
- react-native-svg для donut chart

**Почему:**
- **Expo** даёт единую сборку под web/iOS/Android и быстрый онбординг — `expo install` сам подбирает совместимые версии.
- **Expo Router** — file-based, легко поддерживать структуру табов с кастомным TabBar (FAB-кнопка требует кастомной реализации в любом случае).
- **Zustand** — минимальный boilerplate, прямой доступ через хук, легко мигрировать на Redux/Jotai при росте.
- **Reanimated v4** — единственный способ получить нативно-плавные 60fps анимации на bottom sheets и swipe-to-delete без проседаний.

**Альтернативы, которые отклонили:**
- *Flutter* — отдельный язык/тулчейн, web-сборка хуже.
- *Redux Toolkit* — избыточен для приложения с двумя сущностями (Expense, Category).
- *@gorhom/bottom-sheet* — не работает корректно на Expo Web. Нужно будет писать кастомный шит на Reanimated в любом случае, поэтому смысла подключать gorhom нет.

## 2026-04-25 — Иконки: @expo/vector-icons → набор Feather

**Контекст:** В прототипе все иконки навигации нарисованы инлайн SVG (домик, часы, барчарт, профиль). В React Native инлайн SVG неудобен — для каждой иконки потребуется отдельный компонент.

**Выбор:** `@expo/vector-icons` (идёт в комплекте Expo, не требует установки), набор **Feather** — тонкие однолинейные иконки, визуально совпадающие с прототипом.

**Почему:** Визуально неотличимы от ручных SVG. Использование — `<Feather name="home" size={22} color={color} />`. Нет затрат на ручной перенос.

**Альтернатива:** `react-native-svg` для ручного портирования каждой иконки 1-в-1. Затраты больше, выигрыш — нулевой.

## 2026-04-25 — Удаление трат: swipe-to-delete + Toast Undo

**Контекст:** В прототипе механизма удаления нет.

**Выбор:** Свайп влево по строке → красная подложка с корзиной. Частичный свайп — кнопка "Удалить", полный — удаление сразу. Тост "Удалено · Отменить" внизу на 3.5 сек.

**Где работает:** Главная (Последние траты), История (вся группировка по дням). На Аналитике (Топ-3) — не работает.

**Почему:**
- Свайп — стандартный жест iOS/Android для удаления.
- Tост Undo защищает от случайных удалений без модалки подтверждения (которая раздражает в трекере трат).
- На Аналитике Топ-3 — это аналитический срез, редактирование там нелогично.

**Реализация:** `Gesture.Pan()` из gesture-handler + `useAnimatedStyle` из Reanimated v4. Базовый компонент — `components/ui/SwipeableRow.tsx`, поверх него обёртка `ExpenseRow`.

## 2026-04-25 — iPhone-фрейм только на широком web

**Контекст:** Прототип отображается в декоративном iPhone-корпусе 390×844. Это нужно для отладки в браузере на десктопе. На реальных iOS/Android приложение должно занимать весь экран. Также мобильный Safari/Chrome — это тоже web, и там фрейм выглядит абсурдно («телефон в телефоне»).

**Выбор:** Wrapper `PhoneFrame` рендерит корпус только при `Platform.OS === 'web'` **и** при `window.width >= 600`. На native и на узком web (мобильные браузеры) — passthrough children. Используется `useWindowDimensions()` для реактивного переключения при ресайзе окна.

**Почему 600 px:** выше любого современного телефона в портрете (iPhone 15 Pro Max — 430 logical px), но ниже iPad mini в портрете (768 px). Десктоп / большой планшет получают фрейм, телефоны — нет.

**Почему:** Один и тот же код работает везде, без рантайм-переключений.

## 2026-04-30 — BottomSheet: `createPortal` на web для пробития stacking context

**Контекст:** На web все шиты, открываемые из screen-компонентов (CategoryPickerSheet/SortPickerSheet/DateRangeSheet с History, CategoryEditSheet с Profile), визуально лежали под TabBar. `AddExpenseSheet` работал, потому что был поднят на уровень `app/(tabs)/_layout.tsx` (сиблинг `<Tabs>`). Остальные — внутри screen-ов, которые дети `<Tabs>` → попадали в CSS stacking context, и `zIndex: 400` шита эффективен только в пределах screen-а, не глобально.

**Выбор:** В `BottomSheet.tsx` для `Platform.OS === 'web'` финальное дерево рендерится через `ReactDOM.createPortal(tree, document.body)`. Sheet оказывается прямо в `<body>`, минуя любые stacking context'ы родительских View. На native — обычный in-tree рендер (там `zIndex` через `elevation`/native layering работает иначе и проблемы нет — у нас её и не было).

**Почему Portal, а не lift-state-up:** Альтернатива — перенести все picker-шиты в `_layout.tsx` (как `AddExpenseSheet`) и хранить их `open`-флаги в Zustand-store, чтобы screens могли их открывать. Это ~150 строк рефакторинга через всё приложение и каждое будущее добавление шита. Portal — точечное изменение в 10 строк в одном общем примитиве, покрывает все шиты сразу (и текущие, и будущие).

**Зависимости:** используем `react-dom` (он уже тащится с `react-native-web`). Чтобы не добавлять `@types/react-dom` в package.json (риск drift с React 19), создан локальный shim `types/react-dom.d.ts` с минимальным declaration для `createPortal`. На native импорт `react-dom` не выполняется по run-time (Platform.OS gate), и Metro его не упаковывает в native-bundle.

## 2026-04-28 — Глобальный CSS-сброс через `app/+html.tsx`

**Контекст:** На iOS Safari у `<input>` при фокусе появляется синяя обводка (browser default focus outline). Через RN-Web компонент `TextInput` рендерится в `<input>`, и обводка просачивается в наш UI — выглядит чужеродно.

**Выбор:** Создан `app/+html.tsx` (root HTML template Expo Router) с `<style>`, обнуляющим `outline` и `-webkit-tap-highlight-color` для `input/textarea/select/button`. Туда же вынесен `viewport-fit=cover`.

**Почему глобально, а не точечно по стилям TextInput:** Один файл вместо правок в каждом TextInput. Любые будущие input'ы автоматически без обводки. Минус — выключает outline у всех кнопок, но в этом проекте мы используем кастомные `Pressable` со своими hover/press-состояниями, так что нативный focus-ring нам не нужен.

**Альтернатива (отклонили):** Локальный `outlineStyle: 'none'` в каждом стиле `TextInput`. Дублирование на 3+ компонента, легко забыть в новых местах.

**Дополнения (2026-04-30) — фикс артефактов iOS Safari при открытии клавиатуры:**

К тому же блоку CSS добавлены:
- `width: 100%; overflow-x: hidden;` — запрещают горизонтальный скролл страницы. Без этого iOS Safari при автозуме на input'ы прокручивал страницу вправо: AddExpenseSheet съезжал влево, символ ₽ оказывался за правым краем экрана.
- `height: 100dvh;` (вместо ранее `100%`) — dynamic viewport units учитывают появление/скрытие нижних UI-баров iOS Safari. Не учитывают программную клавиатуру (это известная ограниченность iOS), но без них всё равно рассинхронизация ещё хуже.
- `background-color: #131929` (Colors.surfaceHigh, цвет поверхности bottom sheet'а) вместо `Colors.bg` — на iOS Safari при открытии клавиатуры между нижним краем sheet'а и accessory bar клавиатуры виден полоской body. Закрашивая body в тот же цвет, что и sheet, мы маскируем эту полосу: пользователь видит однородную поверхность от верха sheet до клавиатуры. Когда sheet закрыт, body не виден из-за `appShell` View с `Colors.bg` — побочного эффекта нет.

**Альтернатива (отклонили):** добавить web-only `visualViewport.addEventListener('resize')` в `BottomSheet` для определения высоты клавиатуры и подъёма sheet ровно на её высоту. Это «правильное» решение, но требует JS-обвязки и риск сломать существующий нативный keyboard avoidance. Покраска body визуально решает проблему за одну CSS-строчку и без рисков.

## 2026-04-25 — Supabase: full seed + deferred delete + optimistic без rollback

**Контекст:** Подключаем Supabase. Стек по заданию включает БД, но auth в MVP нет. Решали три развилки.

**Выбор A — Что при пустой БД:** *Full seed.* При первом запуске, если категорий 0, льём 10 дефолтных категорий + 31 mock-трату. Это даёт сразу реалистичный набор для тестирования UI. Mock-данные остаются в репо как seed-источник.

**Выбор B — Mock-fallback:** Сохраняем `buildMockExpenses` и `DEFAULT_CATEGORIES` как fallback. Если Supabase не настроен или fetch упал — приложение работает на моках, не падает. Полезно для разработки без сети.

**Выбор C — Swipe-delete:** *Deferred*. DELETE в БД отправляется только если истёк 3.5с таймер UndoToast'а. Если пользователь нажал "Отменить" — DELETE не отправляется, локального восстановления достаточно. Это даёт 1 запрос вместо 2 при undo и логично (отмена = операция не была подтверждена).

**Дополнительное решение — оптимистические мутации без rollback:** При ошибке push в Supabase локальное состояние НЕ откатывается. Запись остаётся в сторе, показывается `ErrorToast` с описанием. Минус: при следующем `fetchAll` локальное может расходиться с БД. Плюс: пользователь не теряет введённые данные при кратковременном сбое сети.

**Альтернатива (отклонили):** Очередь retry с exponential backoff. Слишком сложно для MVP, добавим в post-MVP.

## 2026-04-25 — Client-side IDs для expenses

**Контекст:** Expenses имеют тип `id: number`. В Supabase поле могло быть `BIGSERIAL` (генерируется БД) или `BIGINT` (генерируется клиентом).

**Выбор:** Клиентская генерация — `Date.now() * 1000 + random(0..999)`.

**Почему:** Optimistic update проще — id известен сразу, не нужно ждать ответа INSERT'а и переписывать ссылки в коллекциях. Коллизии для одного пользователя пренебрежимы (∼1e9 id-пространство в каждой миллисекунде).

**Альтернатива (отклонили):** UUID v4 — потребовал бы менять тип `id: number` на `string`, плюс полифилл `crypto.randomUUID` для нативки.
