## 2026-05-04 — Dynamic categories: переход от slug-таксономии к dynamic + directive parsing

**Контекст:** Изначальное решение (запись от 2026-05-03) — захардкожена 10 slug'ов в системном промпте, клиент маппит slug → реальный `Category.id`. Это работало для дефолтных категорий, но плохо ложилось на реальные сценарии:
- Пользователь добавил «Образование» → модель её не знает → возвращает ближайший `shop`/`other` → юзер всегда жмёт «Изменить»
- Пользователь переименовал «Еда» → «Продукты» → label поменялся, slug `food` остался → семантический mismatch
- Голосовые директивы «запиши в категорию инструменты» физически невозможны без знания списка
- Любая попытка масштабироваться к 15-20 пользовательским категориям ломала всё

**Выбор:** Переход на полностью dynamic — список категорий приходит в каждом request. Промпт описывает **процесс выбора**, а не таксономию.

### Архитектура

| Компонент | Раньше | Теперь |
|---|---|---|
| Системный промпт | Содержит 10 slug'ов + 14 few-shot, привязанных к ним | Описывает формат envelope'а (date + категории + текст), правила выбора. 12 few-shot с **разными** наборами категорий |
| Request body | `{text, today, tz}` | `{text, today, tz, categories: [{id, label, description?}]}` |
| User message модели | `[today: ...]\n<text>` | `[today: ...]\n\nКатегории:\n- food: Еда (продукты, ...)\n- cat_99: Инструменты (фрезер, дрель)\n\nТекст: <text>` |
| Поле `cat` в ответе | Один из 10 жёстких slug'ов или null | Любой `id` из переданного `categories[]` или null |
| Type `ExpenseCat` | Literal union 10 значений | `string` (любой Category.id) |
| Клиентский маппинг slug→id | Был: `categories.find(c => c.id === slug)` | Не нужен — сервер возвращает реальный id |

### Что нового в БД и схеме данных

- **Миграция `0002_category_description.sql`** — добавляет `categories.description TEXT NULL` (применить вручную через Dashboard, как остальные)
- **`Category.description?: string`** — опциональная подсказка для голоса. Дефолтные 10 категорий получают предзаполненные описания (вынесли из промпта в сами категории — `food: 'продукты, Пятёрочка, Магнит'` и т.д.)
- **`backfillDefaultDescriptions(loaded, defaults)`** в `lib/sync.ts` — best-effort заполнение пустых description у дефолтных при первой загрузке после миграции (для существующих установок)

### Directive parsing

Помимо контекстного матчинга («пятёрочка 800» → food), модель теперь понимает явные команды:

1. **Категория**: «в категорию X», «отнеси к X», «запиши в X», «категория X»
   - Найти X в `categories[]` (по label или description)
   - Не нашёл точно → выбрать ближайшую по смыслу из списка (best-effort)
   - Ничего не подходит → null
2. **Имя/комментарий**: «в комментарий X», «напиши в комментарии X», «в название X»
   - `name = X` дословно
3. **Директива всегда важнее контекста.** «в пятёрочке 500 запиши в категорию транспорт» → cat=trans, не food.

### Лимиты

| Что | Лимит | Где enforce |
|---|---|---|
| Кол-во категорий пользователя | 20 | Profile UI: «Добавить» disable'ится |
| Кол-во категорий в request на сервер | 30 | Edge Function отбрасывает с `too_many_categories` 400 |
| Длина description | 200 chars | `MAX_CATEGORY_DESCRIPTION_LEN` в `constants/categories.ts`, UI clamp + counter |
| Длина label на сервере | 60 chars | clamp в `parseCategories()` |

### Цена

Доп. ~120 input-токенов за блок категорий (20 кат × ~6 токенов с label) или ~400 токенов с descriptions. Порядок цены: +2-8 ₽/мес при 100 фразах/день. Несущественно при гранте 4000 ₽.

### Что отказались делать

- **`requestedCategory` в ответе** (для UI-предложения «создать новую категорию X»). YAGNI на сейчас — просто null + ручное создание через Profile. Можно добавить позже.
- **Embeddings-resolver** (двухступенчатый: модель отвечает свободным текстом, клиент через embeddings матчит). Перебор для 15-20 категорий, ещё один источник ошибок и API.

### Где был жёсткий комментарий «не подставляй динамический список»

В `prompt.ts` он перевёрнут на 180° — теперь там описание процесса парсинга envelope'а с динамическим списком. Few-shot переехали на варьирующиеся наборы категорий, чтобы модель училась «брать из переданного списка», а не «помнить эти конкретные slug'и».

## 2026-05-04 — VoiceFab наезжает на TabBar, useTabBarHeight() для padding'а экранов

**Контекст:** После введения floating-pill TabBar и отдельного VoiceFab выявились две UX-проблемы при тесте на iPhone:
1. VoiceFab «парил» над TabBar, занимая отдельное визуальное пространство — выглядело как два независимых элемента, не связанных между собой
2. Контент скроллящихся экранов уходил под floating TabBar — последняя строка списка трат на Главной была наполовину обрезана баром

**Выбор:**

### A. VoiceFab наезжает на TabBar (только нижним краем)
Изначально пробовали полное полу-перекрытие (центр FAB на верхнем крае TabBar, половина внутри/половина снаружи) при размере 60×60 — но при тесте на iPhone это **полностью накрыло иконку профиля** в правом нижнем углу. Поправили в две стороны:

- **Уменьшили VoiceFab до 44×44** (было 60). Теперь он меньше plus-FAB (52) — visually читается как secondary hero, не peer основной навигации.
- **Уменьшили overlap с 30 до 12px.** Только нижний край FAB заходит на TabBar — кнопка «перчится на углу» бара, а не сидит верхом на нём. Иконка профиля под ней остаётся видна (хотя и в нижне-правом углу).

Реализация: `bottom: tabBarHeight - VOICE_FAB_OVERLAP`, где `VOICE_FAB_OVERLAP = 12`. Размер FAB — `VOICE_FAB_SIZE = 44`. Иконка mic / loader внутри — 20px (было 26).

### B. TabBar опущен ближе к низу экрана
`BOTTOM_MARGIN: 12 → 6 → 0`. Бар прижат к safe-area плотно («лежит на дне» вместо парения с воздухом).

**Развязка с VoiceFab.** Изначально mic вычислялся через `tabBarHeight - OVERLAP`, поэтому когда мы опускали бар, mic тоже опускался (и overlap не менялся). После этой итерации mic закреплён на **фиксированной** offset от safe-area-bottom (`VOICE_FAB_BOTTOM_FROM_SAFE_AREA = 50`) — не зависит от BOTTOM_MARGIN бара. Теперь опускание бара даёт желаемый эффект «mic визуально торчит сильнее над баром», а не двигается синхронно с ним.

### C. `useTabBarHeight()` для padding'а скроллящихся экранов
Раньше каждый экран жил «не зная» о floating-баре, контент скроллился под ним. Теперь экспортирован hook `useTabBarHeight()` из `components/navigation/TabBar.tsx`, возвращающий полную высоту бара (safe-area + bottom margin + visible pill height). Все 4 экрана (`index`, `history`, `analytics`, `profile`) применяют его как `paddingBottom + breathing room` к своему scroll-контейнеру.

**Особый случай — Главная.** Там используется adaptive count «последних трат» через `onLayout`. Без paddingBottom recent-list flex-grow'ом расширялся под TabBar, и onLayout возвращал высоту, в которую помещалась лишняя строка → пользователь видел половину обрезанной строки внизу. С paddingBottom recent-list ограничен до видимой зоны, adaptive count корректный, обрезанных строк больше нет.

**Альтернативы (отклонили):**
- *Сделать каждый экран скроллящимся.* Главная by design не скроллится — это прописано в UX-намерении (быстрый обзор без скролла). Простое добавление паддинга решает проблему без нарушения этого принципа.
- *Хардкодить высоту TabBar в каждом экране.* Дублирование, плюс safe-area меняется по устройствам — централизованный hook даёт правильное значение всегда.
- *Использовать context вместо hook.* Hook чище — нет необходимости в Provider'е, и `useSafeAreaInsets()` уже используется как hook (тот же паттерн).

### D. Экспорт из TabBar
- `TAB_BAR_SIDE_MARGIN` — для VoiceFab расположения (right: SIDE_MARGIN + 4)
- `useTabBarHeight()` — для paddingBottom на screens

Чтобы избежать дублирования значений между `TabBar.tsx` и `_layout.tsx` (раньше было 3 константы в обоих файлах).

## 2026-05-04 — TabBar pill-floating + blur, VoiceFab отделён, иконки без лейблов

**Контекст:** После переезда mic'а в TabBar (запись ниже от 2026-05-04) бар стал визуально перегружен — два FAB по 52px и четыре таба с лейблами под иконками. На реальном iPhone тесно, лейблы дублируют узнаваемые иконки, плотность давит.

**Выбор:** Комбо из трёх изменений, применённых вместе:

### A. Убрали текстовые лейблы под иконками табов
Современный iOS-паттерн (Instagram, Twitter/X, Threads). Иконки `home / clock / bar-chart-2 / user` достаточно узнаваемы, активный таб подсвечивается accent-цветом + pill-фоном `accent @ 14%` под иконкой. Иконки увеличены 22 → 24px. Освободило ~16px вертикали и убрало визуальный шум.

### B. TabBar — pill-shaped floating
Бар отделён от низа экрана: `marginHorizontal: 16, marginBottom: 12 + safeAreaBottom`. `borderRadius: 36` — pill-форма. Background — `BlurView` (`expo-blur`, intensity 60, tint dark) поверх лёгкого `Colors.surface @ 40%` тинта (на web heavier 70% — там backdrop-filter выходит прозрачнее). Тонкая обводка `#ffffff @ 8%` + drop shadow `0 12px 28px rgba(0,0,0,0.55)` дают эффект «острова» над контентом. Бар позиционируется `position: absolute` — экраны скроллятся под ним без принудительного bottom-padding.

**Зависимости:** `expo-blur ~15.0.8` (новый config-plugin, добавлен в package.json). На native — настоящий iOS-blur через UIVisualEffectView. На web — fallback на CSS `backdrop-filter: blur()`, поддерживается всеми современными браузерами.

### C. Mic вынесен в собственный VoiceFab
Микрофон полностью убран из TabBar в самостоятельный компонент `components/navigation/VoiceFab.tsx`. Расположение — bottom-right thumb zone, `right: 20, bottom: tabBarTopOffset + 8` (над TabBar). Размер 60×60 — чуть крупнее обычных табных FAB, чтобы голос читался как hero-кнопка. Все ripple- и processing-анимации переехали в этот компонент. TabBar теперь чисто навигационный + один центральный «+» FAB для ручного ввода.

`VoiceFab` принимает пропс `visible: boolean` — `app/(tabs)/_layout.tsx` ставит `false` пока открыт `AddExpenseSheet` или `VoiceConfirmSheet`, чтобы кнопка не торчала над шитом и не путала layering.

**Почему именно эта раскладка (mic правый-низ, не правый-верх):**
- В thumb zone большого пальца, как Material FAB и iOS-стандартный паттерн для primary action
- Близко к TabBar — голос продолжает быть «нижней» зоной взаимодействия рядом с навигацией и плюсом
- Топ-бара как такового в приложении нет (на каждом экране свой header), правый-верх требовал бы отдельной обработки на каждом экране

**Альтернативы (отклонили):**
- *Только B (pill+blur), оставить mic в TabBar.* Не решает плотности — два FAB всё равно тесные внутри pill.
- *Поднять FAB сильнее над TabBar (`translateY: -24`).* Эффект «hero», но кнопки отрываются от бара визуально и асимметрия одного-внутри + одного-сверху смотрится странно.
- *Не делать blur, остаться на solid surface.* Pill без blur — обычный rounded bottom bar, без премиум-эффекта.

### Документация
Обновлены `components/navigation/README.md` (TabBar API убавился, добавился VoiceFab), `docs/voice-input.md` (sequence-диаграмма с VoiceFab вместо mic-FAB-в-баре), эта запись.

## 2026-05-04 — Микрофон вынесен в TabBar как второй FAB, confirm — отдельный шит

**Контекст:** В первой версии голос жил внутри `AddExpenseSheet` — иконка микрофона стояла рядом с полем суммы, нажатие требовало предварительного открытия формы. После теста стало ясно: лишний шаг (тап на «+» → дождаться открытия → тап на mic) ломает основной сценарий «один жест — голосом ввёл трату». Плюс на iOS клавиатура схлопывалась/перерисовывалась при переходе формы в recording-режим, давая визуальный мусор и неудобный UX.

**Выбор:** Голос полностью вынесен из `AddExpenseSheet` на верхний уровень — иконка микрофона стала вторым FAB-кругом в `TabBar`, рядом с «+». Оба круга одинакового размера и веса, потому что голос — основная фича. Подтверждение распознанного теперь в собственном шите `VoiceConfirmSheet`.

**Что поменялось:**

| Что | Было | Стало |
|---|---|---|
| Иконка mic | Внутри AddExpenseSheet | Второй FAB в TabBar (рядом с «+») |
| Состояние записи | `voiceState` в AddExpenseSheet | `voiceState` в `app/(tabs)/_layout.tsx` |
| Confirm-карточка | Заменяла поля формы внутри AddExpenseSheet | Отдельный `VoiceConfirmSheet` |
| «Изменить» в confirm | Заполнял поля прямо в AddExpenseSheet | Кладёт ParsedExpense в `useStore.voicePending` → закрывает confirm → открывает AddExpenseSheet, который читает voicePending на open и предзаполняет |
| Toast «Только в приложении на iPhone» / «Не удалось распознать» | Внутри AddExpenseSheet через topBanner | Над TabBar, в `_layout.tsx` |
| TabBar | 4 таба + один FAB по центру | 4 таба + центральный слот с двумя FAB (`flex: 2` для слота, mic+plus с гэпом 14px) |

**Архитектурные нюансы:**

- **`useStore.voicePending`** — узкоспециализированный handoff slot. Когда пользователь жмёт «Изменить» в confirm-шите, мы НЕ можем напрямую передать `pending` в `AddExpenseSheet` (это два независимых шита с разной семантикой open/close). Прокидывать через пропсы из `_layout.tsx` неудобно — добавлять `initialValues?: ParsedExpense` в AddExpense означает связать его с voice. Через store решение чище: одно поле, set-в-confirm, read-в-add, clear-в-add. Никто кроме этих двух шитов на него не смотрит.
- **AddExpenseSheet больше не знает про голос.** Никаких импортов из `lib/voice`, никаких state-машин, никаких toast'ов. Только чтение `voicePending` на open для предзаполнения. Это возвращает компонент к одной ответственности — ввод траты руками.
- **VoiceConfirmSheet тоже ничего не знает про state-машину** — это чистый UI-компонент, рисует карточку и зовёт `onSave` / `onEdit`. Вся механика recording/processing/error живёт в `_layout.tsx`.
- **Push-to-talk на TabBar** — `Pressable` с `onPressIn` / `onPressOut` (раньше тоже были эти колбэки, но через AddExpenseSheet). Сейчас они летят прямо в `_layout.tsx` через пропсы TabBar. Race-fix через `pressedRef` сохранён.
- **TabBar API** — расширен 4 пропсами (`onMicPressIn`, `onMicPressOut`, `micRecording`, `micProcessing`). На native + web — одинаковый набор, mic-FAB одинаковый. Web-сценарий обработан в `_layout.tsx`: `Platform.OS === 'web'` → toast вместо записи.

**Альтернатива (отклонили):** хранить весь voice-state в Zustand (включая recording/processing). Перебор — это локальный UI-state одного экрана, а не «доменное состояние приложения». Стор раздулся бы без пользы. Из стора нужен только `voicePending` для handoff'а между двумя шитами.

**Документация:** обновлены `components/navigation/README.md`, `components/sheets/README.md`, `store/README.md`, `lib/voice/README.md`, `docs/voice-input.md`. Картинка sequence-диаграммы в `voice-input.md` переведена на новую схему «mic в TabBar → VoiceConfirmSheet → AddExpenseSheet (опционально для Изменить)».

## 2026-05-04 — Голос на web не поддерживаем, дата в NLP-парсинге, BottomSheet portal внутрь PhoneFrame

Три связанных решения после первого живого теста голосового ввода. Объединил в одну запись, потому что они появились в одной итерации полировки.

### A. Голосовой ввод на web — официально не поддерживаем

**Контекст:** Native (Expo Go) flow заработал чисто, web (Safari/Chrome) — упёрся в формат аудио. Yandex SpeechKit synchronous API принимает только `lpcm` (raw PCM) и `oggopus` (opus в OGG-контейнере). Что отдают браузеры:
- **Chrome/Edge:** `audio/webm;codecs=opus` — opus-кодек, но в WebM-контейнере. SpeechKit видит первые байты, не находит OggS-header → 400 «ogg header has not been found».
- **Firefox:** `audio/ogg;codecs=opus` — теоретически бы прошло, но MediaRecorder.isTypeSupported в порядке предпочтения возвращает webm первым.
- **Safari (desktop и iOS):** `audio/mp4` (AAC) — SpeechKit вообще не понимает.

**Варианты, которые рассматривали:**
- **A1.** Поменять приоритет MIME (ogg first) → решает только Firefox, Chrome всё равно сломан.
- **A2.** Серверный ремукс webm→ogg в `transcribe-audio` → решает Chrome/Firefox/Edge. Реально 150 строк кода: opus-frames из webm-контейнера → запаковка в ogg-контейнер. Кодек не трогается, только wrapper. Safari всё равно не работает (mp4/AAC — нужен полноценный декодер).
- **A3.** WebAssembly ffmpeg на клиенте → +5 МБ к web-бандлу.
- **A4.** Async SpeechKit API → принимает webm, но требует polling, ~30 мин работы.

**Выбор:** Ни один из A1–A4. Web-голос откладываем как known limitation, фокусируемся на native (главный канал). UX-fix: при нажатии mic на web — короткий toast «Только в приложении на iPhone». Web-recorder (`lib/voice/recorder.impl.ts`, `wavStrip.ts`, `base64.ts`) остаётся в коде на будущее — он валиден, просто не вызывается.

**Почему именно эта развилка:** Для личного проекта, где основной канал тестирования — Expo Go, и для друзей в будущем будет TestFlight / App Store, web был приятным бонусом, не блокером. Два часа на ремукс ради бонуса — невыгодно. В roadmap зафиксирован follow-up для сценария «когда web-аудитория станет нужна».

### B. Дата в NLP-парсинге

**Контекст:** Изначальный `ParsedExpense` был `{amount, name, cat}` — дата всегда «сегодня». Это режет реальный сценарий: «вчера обед 500», «25 апреля такси 350», «3 дня назад продукты 1200». Голосовой ввод теряет смысл если нельзя ввести в день, отличный от текущего.

**Выбор:** Расширили `ParsedExpense` четвёртым полем `date: string | null` в формате `YYYY-MM-DD` (локальный таймзонт пользователя).

**Реализация:**

| Слой | Что |
|---|---|
| Клиент (`parseExpense.ts`) | Передаёт в request body два дополнительных поля: `today` (YYYY-MM-DD из `new Date()` локально, не UTC) и `tz` (`Intl.DateTimeFormat().resolvedOptions().timeZone`) |
| Сервер (`index.ts`) | Подставляет `today` и `tz` в начало user-message: `[today: 2026-05-04, tz: Europe/Moscow]\n<phrase>`. Сам системный промпт ничего не вычисляет — модель сама считает «вчера = today − 1» от переданного значения |
| Промпт (`prompt.ts`) | Расширены правила (5 новых пунктов про даты) и добавлены 4 few-shot с конкретными датами относительно today=2026-05-04 для иллюстрации |
| Валидатор (`validate.ts`) | Регэксп `^\d{4}-\d{2}-\d{2}$` + проверка реальности даты через round-trip + sanity window: не более чем today + 1 день (TZ-погрешность) и не дальше today − 365 дней. Любое нарушение → `null`, без exception |
| UI (`AddExpenseSheet.tsx`) | Confirm-карточка показывает дату строкой «Вчера», «3 дня назад», «5 апреля» (через существующий `dayHeader` из `utils/format.ts`), но только если она ≠ today. «Сохранить» использует `pending.date`, «Изменить» заполняет `setDate` |

**Почему именно так:**
- **Today передаёт клиент**, не сервер сам считает. Сервер не знает в каком таймзонте пользователь, и Frankfurt-edge вокруг полуночи может быть не в том дне.
- **Дата только в формате YYYY-MM-DD**, без времени. Голосом всё равно никто не диктует «обед в 14:23», нужен только день. Время в `Expense.date` ставится 00:00 локально — для группировки по дням.
- **Лимит 365 дней назад**, не 5 лет. 99% реальных голосовых вводов — про сегодня/вчера/недавно. Если кто-то реально захочет ввести трату годовалой давности — это «Изменить» в форме, обычный ввод даты через DrumPicker.
- **Если модель вернула «завтра»** — отбрасываем (не бывает трат в будущем).
- **Если today/tz не переданы клиентом** — модель парсит только абсолютные даты («25 апреля»), относительные → `null`. Это graceful degradation.

### C. BottomSheet portal target внутри PhoneFrame

**Контекст:** Запись от 2026-04-30 («BottomSheet: createPortal на web для пробития stacking context») — все шиты на web портятся в `document.body`. Это решило проблему перекрытия TabBar, но создало новую — на десктопе шит покрывал весь монитор, выходя за границы декоративного PhoneFrame (390×844).

**Выбор:** `PhoneFrame` теперь рендерит внутри себя `<View>` через `<PhoneFramePortalHost>`, его DOM-элемент кладётся в React Context (`PhoneFramePortalContext`). `BottomSheet` читает Context — если есть target, портит туда; иначе fallback на `document.body`.

**Почему так:**
- Stacking context всё ещё пробивается (шит минует Tabs/Screens children-stack), потому что portal по своей сути выпрыгивает из дерева React, а DOM-target по-прежнему вне Tabs-разметки.
- Sheet остаётся **внутри** PhoneFrame визуально — потому что target — это absolute-positioned div *внутри* phone-inner.
- Mobile web (узкий, без PhoneFrame) — fallback на body, шит покрывает весь viewport (как и должен на телефоне).
- Native — `Platform.OS !== 'web'` гейт, host-View не рендерится, Context даёт `el = null`, BottomSheet рендерит in-tree (как раньше).

**Альтернатива (отклонили):** хранить target как глобальную переменную (`window.__phonePortal`). Работает, но плохой паттерн — тестировать сложнее, race на mount/unmount.

## 2026-05-03 — STT через Yandex SpeechKit (через Edge Function), confirm-карточка как UX-pattern

**Контекст:** К NLP-парсингу через `parse-expense` (запись от 2026-05-03 ниже) добавляем второй шаг — собственно голос → текст. Пользователь нажимает иконку микрофона в `AddExpenseSheet`, диктует фразу, отпускает, подтверждает результат.

**Развилка по STT-провайдеру (3 варианта):**

| Вариант | Plus | Minus |
|---|---|---|
| A. `expo-speech-recognition` на native + Web Speech API на web | live-транскрипт, on-device на iOS, бесплатно | требует dev client (вылет из Expo Go) |
| B. **Yandex SpeechKit через Edge Function-прокси** (выбрано) | работает в Expo Go, единая архитектура с GPT, секреты только на сервере | задержка ~1.5 сек (нет live), цена ~0.18 ₽/фразу |
| C. Гибрид (Web Speech API на web + SpeechKit на native) | сохраняем Expo Go | два пути → лишняя сложность |

**Выбор — B.** Причины:
1. Главный канал тестирования — Expo Go на iPhone. Переход на dev client — лишняя операция и лишние 30 минут на новой машине.
2. Нет live-транскрипта — не блокер. UX «зажал → сказал → отпустил → подтвердил» не подразумевает что пользователь смотрит на иконку во время речи. 1.5 сек ожидания после отпускания закрываем spinner'ом.
3. Симметрия архитектуры: STT и NLP — обе через Edge Function, оба с одним ключом, одни логи, одна точка отказа.
4. Цена ничтожна: 100 фраз/день × 0.48 ₽ = 48 ₽/мес. Грант 4000 ₽ Yandex Cloud покроет на годы.

Когда захотим live-транскрипт — мигрируем на A одним рефакторингом native-recorder.

**Дополнительные решения:**

- **Push-to-talk, не tap-to-toggle.** Зажал палец = пишем. Отпустил = стоп. Это устраняет догадку «закончил или думает», и не нужен voice activity detection.
- **Confirm-карточка после распознавания.** Не показываем сырой STT-текст. Показываем только итоговый матч после NLP — крупная сумма, chip категории с цветом, имя. Кнопки «Изменить» / «Сохранить». Это меньше визуального шума и явный UX-фокус на «правильно ли я тебя понял», а не «всё ли я расслышал».
- **«Изменить» = заполнить поля формы распознанным**, не «откатить к тому что было». Иначе теряется работа модели — пользователь решил «не идеально, поправлю», логично сохранить распознанное в форме.
- **Голос полностью переписывает уже введённые в форме значения.** Если пользователь вручную ввёл 1500, потом сказал «такси 350» — confirm покажет 350. Намерение — заменить.
- **Защитные пороги:** MIN 0.5 сек (короче — игнорируем), MAX 15 сек (auto-stop), MAX 256 КБ на сервере. Если STT/NLP вернули all-null — warning toast «Не удалось распознать», без confirm-карточки.

**Аудио-форматы:**

| Платформа | Решение |
|---|---|
| iOS native | LPCM 16 кГц mono `.wav` через `expo-audio`; на клиенте `wavStrip.ts` режет 44-байтный RIFF-header → raw PCM → `audio/x-pcm` → SpeechKit `lpcm` |
| Web | `MediaRecorder` `audio/webm;codecs=opus` → SpeechKit `oggopus` (битстрим webm/opus совместим с oggopus) |

iOS Safari может отдать `audio/mp4` вместо ogg/opus — SpeechKit его в sync-режиме не принимает. Документировано в `docs/voice-input.md` как известное ограничение для старых iOS Safari. Не блокер — на iOS 17+ Safari отдаёт ogg/opus.

**Транспорт base64-в-JSON, не multipart.** `supabase-js` `functions.invoke()` сериализует body как JSON по дефолту. Прицепить Blob через multipart на React Native неудобно. Base64 даёт +33% к размеру (3 КБ → 4 КБ на секунду opus) — несущественно при cap'е 256 КБ.

**Что нового в репо:**
- `supabase/functions/transcribe-audio/` — новая Edge Function
- `supabase/functions/_shared/cors.ts` — общий CORS-хелпер (рефакторинг — вынес из `parse-expense/cors.ts`)
- `lib/voice/` — клиентский слой (recorder, transcribe, captureExpense, base64, wavStrip)
- `expo-audio` ~1.1.x — новая зависимость (config-plugin)
- `app.json` — `ios.infoPlist.NSMicrophoneUsageDescription`, `android.permissions: ['RECORD_AUDIO']`
- `docs/voice-input.md` — сквозная документация фичи

## 2026-05-03 — NLP-парсинг трат: YandexGPT через Edge Function-прокси, slug-таксономия

**Контекст:** Готовим голосовой ввод траты — пользователь говорит «в пятёрочке на 800», приложение должно заполнить форму (сумма, название, категория). NLP-часть — отдельный шаг от STT: на вход — строка, на выход — `{amount, name, cat}`. Решали три развилки.

**Выбор A — Где живёт ключ:** *Edge Function-прокси, не клиент.* YandexGPT API-key — биллинговый секрет. Если положить его в клиентский бандл (`EXPO_PUBLIC_*` или хардкод), любой посетитель https://moneyspender.vercel.app откроет devtools и выкачает ключ → начнёт тратить квоту в Yandex Cloud. Это принципиально отличается от Supabase anon-key (который публичный по дизайну, см. запись от 2026-04-27). Поэтому ключ хранится в Supabase secrets, клиент ходит через `supabase.functions.invoke('parse-expense')`. Цена решения — +1 сетевой хоп (~200мс) и зависимость от Supabase CLI для деплоя.

**Выбор B — Какая модель:** *YandexGPT-lite.* Парсинг короткой фразы — простая задача, lite-модель справляется при `temperature: 0` и хорошем few-shot. Цена — порядка 0.30 ₽ за вызов, против ~3 ₽ у full-модели. Альтернативы (отклонили):
- *OpenAI / Anthropic* — лучшее качество русского, но требует ключ + VPN с РФ-инфры Supabase (Frankfurt).
- *Локальная LLM на устройстве* — слишком тяжёлая для Expo Go и web.

**Выбор C — Как передавать список категорий модели:** *Захардкоженная slug-таксономия в промпте, не динамический список пользователя.*

Альтернатива была — каждый запрос подкладывать в промпт реальный `categories` из стора, чтобы модель видела актуальные id и названия после правок пользователя на Профиле. Отклонили по четырём причинам:

1. **Стабильность few-shot.** Примеры в промпте привязаны к slug'ам (`food/cafe/...`). Менять набор без переобучения примеров — теряем точность.
2. **Резистентность к переименованиям.** Пользователь переименовал «Еда» → «Продукты». Slug `food` не должен от этого меняться — иначе сегодня модель отвечает `food`, завтра `продукты`, и UI-маппинг ломается. С фиксированной таксономией: модель всегда возвращает `food`, клиент маппит `food → catId` пользователя, а как у пользователя называется эта категория — модели всё равно.
3. **Резистентность к удалениям.** Удалил категорию «Спорт» — модель всё равно может вернуть slug `sport` (он в её таксономии). Клиент в этом случае откатывается на `other` или на текущую выбранную в форме категорию.
4. **Толерантность к новым категориям.** Пользователь добавил свою «Образование» — модель её не предложит, но это и не страшно: голосовой ввод заполнит сумму и название, а категорию пользователь выберет сам в форме.

Маппинг slug → `Category.id` живёт на стороне вызывающего компонента (типично `AddExpenseSheet`). По счастью slug'и таксономии 1:1 совпадают с `id` дефолтных 10 категорий из `constants/categories.ts` — это конвенция, не требование. Если пользователь переименовал/удалил, маппинг должен обработать оба случая.

В `prompt.ts` стоит крупный комментарий-предупреждение для будущего разработчика: «не подставлять сюда динамический список — это таксономия, не данные».

**Дополнительные решения:**

- **`maxTokens: 100`** — короткий ответ-JSON. Дороже не имеет смысла, опаснее (модель начнёт болтать).
- **Хард-кап `MAX_INPUT_LEN = 500`** на сервере — защита от prompt-injection через гигантские payload'ы и от случайных биллинг-выстрелов.
- **`stream: false`** — клиенту не нужен стриминг, ответ маленький.
- **All-null ответ — не ошибка.** Если модель не справилась — клиент просто не заполняет форму (без тоста-ошибки). Тост показываем только при сетевых сбоях/HTTP-ошибках функции.
- **Без retry-очереди.** Голосовой ввод интерактивный, пользователь сразу видит провал и переговаривает. Очередь только задержит фидбек.

**Где хранятся секреты:** только в Supabase secrets (`supabase secrets set YANDEX_API_KEY=... YANDEX_FOLDER_ID=...`). В репо — **никогда**. Локальная разработка через `supabase functions serve --env-file .env.local`, файл в `.gitignore`.

## 2026-04-28 — История: иконки сортировки/поиска в правом углу + фильтр диапазона

**Контекст:** В MVP-версии Истории были два pill'а — «Категория» (фильтр) и «Дата» (на самом деле — сортировка `по дате/по сумме`). Имя «Дата» сбивало с толку. Также не было возможности отфильтровать траты по диапазону дат.

**Выбор:**
- В правом верхнем углу — две иконки: 🔍 поиск + ↕ сортировка. Иконка сортировки меняет вид (clock / bar-chart-2) в зависимости от выбранного режима. Активный режим подсвечен accent-цветом.
- Pill «Дата» убран. На его месте — pill «Период» (фильтр по диапазону дат), открывает новый `DateRangeSheet`.
- Поиск разворачивается inline-строкой под фильтрами (TextInput с `autoFocus`). Live-фильтр по `expense.name` (case-insensitive). При закрытии query очищается.
- Порядок применения фильтров: категория → диапазон дат → поисковый запрос → сортировка.

**Почему поиск inline, а не в шите:** Поиск — частая операция, не должен прятаться за двумя тапами. Inline-строка с автофокусом даёт мгновенный ввод.

**Почему диапазон дат — отдельный шит, а не два инпута:** Drum-пикер (как у `DatePickerSheet`) — наш единый паттерн ввода даты. Два drum'а друг под другом дают сразу обе границы. Кнопка «Сбросить» очищает оба значения.

**Альтернатива (отклонили):** иконка фильтра-всё-в-одном с шитом из 4 секций (категория/период/сортировка/поиск). Слишком много действий за одним жестом, теряется UX-разделение «вижу что фильтрую» от «как сортирую».

## 2026-04-28 — Drill-down с чартов: tap по сегменту/бару → История

**Контекст:** Чарты `BarChart` и `DonutChart` были чисто декоративные. Аналитика и Главная не давали способа быстро провалиться в траты конкретной категории.

**Выбор:** У обоих чартов добавлен опциональный проп `onSelect(catId)`. Если передан — каждый бар/сегмент становится кликабельным:
- **Хаптика**: `lightTap()` на press-in.
- **Pulse-анимация**: бар — scale 1 → 1.06 → 1; сегмент — strokeWidth 22 → 28 → 22. Easing — тот же `cubic-bezier(0.16, 1, 0.3, 1)`, что и у анимации появления.
- **Навигация**: `setTimeout(140 ms)` — даёт пользователю увидеть pulse, после чего `router.push('/history', { cat })`.

В Истории: `useLocalSearchParams<{ cat?: string }>()` + `useEffect` ставит `filterCat` и сразу вызывает `router.setParams({ cat: undefined })`, чтобы при ручном переключении табов фильтр не возвращался.

**DonutChart — два способа тапа по сегменту:**
1. **Hit-test по SVG**: на `Pressable` поверх `<Svg>` ловим `locationX/Y`, считаем угол `atan2` от центра, ищем slice по диапазону `[startAngle, endAngle]`. Корректно учитывается `rotate(-90)` (SVG starts at 12 o'clock).
2. **Тап по легенде**: каждый ряд легенды обёрнут в `Pressable`. Запускает ту же pulse-анимацию сегмента и навигацию.

**Почему оба способа:** Hit-test даёт «настоящий» тап по сегменту, как пользователь ожидает. Легенда — резервный способ для маленьких сегментов, куда трудно попасть пальцем (< 5%). Реализация дешёвая — оба используют одну и ту же функцию `triggerSlice(id)`.

**Альтернатива (отклонили):** только тап по легенде (D2). Проще, но формально не решает задачу «тап по сегменту» из требования.

## 2026-04-25 — iPhone-фрейм только на широком web

**Контекст:** Прототип отображается в декоративном iPhone-корпусе 390×844. Это нужно для отладки в браузере на десктопе. На реальных iOS/Android приложение должно занимать весь экран. Также мобильный Safari/Chrome — это тоже web, и там фрейм выглядит абсурдно («телефон в телефоне»).

**Выбор:** Wrapper `PhoneFrame` рендерит корпус только при `Platform.OS === 'web'` **и** при `window.width >= 600`. На native и на узком web (мобильные браузеры) — passthrough children. Используется `useWindowDimensions()` для реактивного переключения при ресайзе окна.

**Почему 600 px:** выше любого современного телефона в портрете (iPhone 15 Pro Max — 430 logical px), но ниже iPad mini в портрете (768 px). Десктоп / большой планшет получают фрейм, телефоны — нет.

**Почему:** Один и тот же код работает везде, без рантайм-переключений.

## 2026-04-30 — BottomSheet: `createPortal` на web для пробития stacking context

**Контекст:** На web все шиты, открываемые из screen-компонентов (CategoryPickerSheet/SortPickerSheet/DateRangeSheet с History, CategoryEditSheet с Profile), визуально лежали под TabBar. `AddExpenseSheet` работал, потому что был поднят на уровень `app/(tabs)/_layout.tsx` (сиблинг `<Tabs>`). Остальные — внутри screen-ов, которые дети `<Tabs>` → попадали в CSS stacking context, и `zIndex: 400` шита эффективен только в пределах screen-а, не глобально.

**Выбор:** В `BottomSheet.tsx` для `Platform.OS === 'web'` финальное дерево рендерится через `ReactDOM.createPortal(tree, document.body)`. Sheet оказывается прямо в `<body>`, минуя любые stacking context'ы родительских View. На native — обычный in-tree рендер (там `zIndex` через `elevation`/native layering работает иначе и проблемы нет — у нас её и не было).

**Почему Portal, а не lift-state-up:** Альтернатива — перенести все picker-шиты в `_layout.tsx` (как `AddExpenseSheet`) и хранить их `open`-флаги в Zustand-store, чтобы screens могли их открывать. Это ~150 строк рефакторинга через всё приложение и каждое будущее добавление шита. Portal — точечное изменение в 10 строк в одном общем примитиве, покрывает все шиты сразу (и текущие, и будущие).

**Зависимости:** используем `react-dom` (он уже тащится с `react-native-web`). Чтобы не добавлять `@types/react-dom` в package.json (риск drift с React 19), создан локальный shim `types/react-dom.d.ts` с минимальным declaration для `createPortal`. На native импорт `react-dom` не выполняется по run-time (Platform.OS gate), и Metro его не упаковывает в native-bundle.

## 2026-04-28 — Глобальный CSS-сброс через `app/+html.tsx`

**Контекст:** На iOS Safari у `<input>` при фокусе появляется синяя обводка (browser default focus outline). Через RN-Web компонент `TextInput` рендерится в `<input>`, и обводка просачивается в наш UI — выглядит чужеродно.

**Выбор:** Создан `app/+html.tsx` (root HTML template Expo Router) с `<style>`, обнуляющим `outline` и `-webkit-tap-highlight-color` для `input/textarea/select/button`. Туда же вынесен `viewport-fit=cover`.

**Почему глобально, а не точечно по стилям TextInput:** Один файл вместо правок в каждом TextInput. Любые будущие input'ы автоматически без обводки. Минус — выключает outline у всех кнопок, но в этом проекте мы используем кастомные `Pressable` со своими hover/press-состояниями, так что нативный focus-ring нам не нужен.

**Альтернатива (отклонили):** Локальный `outlineStyle: 'none'` в каждом стиле `TextInput`. Дублирование на 3+ компонента, легко забыть в новых местах.

**Дополнения (2026-04-30) — фикс артефактов iOS Safari при открытии клавиатуры:**

К тому же блоку CSS добавлены:
- `width: 100%; overflow-x: hidden;` — запрещают горизонтальный скролл страницы. Без этого iOS Safari при автозуме на input'ы прокручивал страницу вправо: AddExpenseSheet съезжал влево, символ ₽ оказывался за правым краем экрана.
- `height: 100dvh;` (вместо ранее `100%`) — dynamic viewport units учитывают появление/скрытие нижних UI-баров iOS Safari. Не учитывают программную клавиатуру (это известная ограниченность iOS), но без них всё равно рассинхронизация ещё хуже.
- `background-color: #131929` (Colors.surfaceHigh, цвет поверхности bottom sheet'а) вместо `Colors.bg` — на iOS Safari при открытии клавиатуры между нижним краем sheet'а и accessory bar клавиатуры виден полоской body. Закрашивая body в тот же цвет, что и sheet, мы маскируем эту полосу: пользователь видит однородную поверхность от верха sheet до клавиатуры. Когда sheet закрыт, body не виден из-за `appShell` View с `Colors.bg` — побочного эффекта нет.

**Альтернатива (отклонили):** добавить web-only `visualViewport.addEventListener('resize')` в `BottomSheet` для определения высоты клавиатуры и подъёма sheet ровно на её высоту. Это «правильное» решение, но требует JS-обвязки и риск сломать существующий нативный keyboard avoidance. Покраска body визуально решает проблему за одну CSS-строчку и без рисков.

## 2026-04-25 — Supabase: full seed + deferred delete + optimistic без rollback

**Контекст:** Подключаем Supabase. Стек по заданию включает БД, но auth в MVP нет. Решали три развилки.

**Выбор A — Что при пустой БД:** *Full seed.* При первом запуске, если категорий 0, льём 10 дефолтных категорий + 31 mock-трату. Это даёт сразу реалистичный набор для тестирования UI. Mock-данные остаются в репо как seed-источник.

**Выбор B — Mock-fallback:** Сохраняем `buildMockExpenses` и `DEFAULT_CATEGORIES` как fallback. Если Supabase не настроен или fetch упал — приложение работает на моках, не падает. Полезно для разработки без сети.

**Выбор C — Swipe-delete:** *Deferred*. DELETE в БД отправляется только если истёк 3.5с таймер UndoToast'а. Если пользователь нажал "Отменить" — DELETE не отправляется, локального восстановления достаточно. Это даёт 1 запрос вместо 2 при undo и логично (отмена = операция не была подтверждена).

**Дополнительное решение — оптимистические мутации без rollback:** При ошибке push в Supabase локальное состояние НЕ откатывается. Запись остаётся в сторе, показывается `ErrorToast` с описанием. Минус: при следующем `fetchAll` локальное может расходиться с БД. Плюс: пользователь не теряет введённые данные при кратковременном сбое сети.

**Альтернатива (отклонили):** Очередь retry с exponential backoff. Слишком сложно для MVP, добавим в post-MVP.

## 2026-04-25 — Client-side IDs для expenses

**Контекст:** Expenses имеют тип `id: number`. В Supabase поле могло быть `BIGSERIAL` (генерируется БД) или `BIGINT` (генерируется клиентом).

**Выбор:** Клиентская генерация — `Date.now() * 1000 + random(0..999)`.

**Почему:** Optimistic update проще — id известен сразу, не нужно ждать ответа INSERT'а и переписывать ссылки в коллекциях. Коллизии для одного пользователя пренебрежимы (∼1e9 id-пространство в каждой миллисекунде).

**Альтернатива (отклонили):** UUID v4 — потребовал бы менять тип `id: number` на `string`, плюс полифилл `crypto.randomUUID` для нативки.
